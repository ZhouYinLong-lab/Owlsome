# AI 交互式数学学习平台项目书（初稿）

## 1. 项目背景

本项目计划参加 EL 大赛交互组，目标是围绕“AI + 教育 + 交互式学习”做一个可落地的数学学习平台。

传统数学学习资料通常分散在教材 PDF、教师课件、学生笔记、题库、解析文档等不同来源中。学生在学习时经常面临资料难以整理、知识点结构不清晰、例题与讲解分散、做题反馈不足、复习路径不明确等问题。

本项目希望通过 PDF 解析、Markdown 清洗、LLM 内容理解与结构化整理，将原本静态、分散的数学资料重构成可交互、可复用、可持续补充的学习系统。

项目不是单纯做电子教材，也不是只做一个问答机器人，而是做一套从资料解析、内容拆解、学习路径生成到交互学习的完整流程。

## 2. 项目定位

项目名称暂定：AI 交互式数学学习平台

参赛方向：交互组

核心目标：

- 将教材、讲义、笔记、题库等资料自动转化为结构化学习单元。
- 支持学生自主上传 PDF，生成个人学习空间。
- 支持笔记上传、审核与合并，逐步建设可复用的课程知识库。
- 通过 AI 讲解、例题解析、练习生成和学习反馈，形成完整学习闭环。

一句话概括：

> 把静态数学资料变成可拆解、可交互、可持续更新的 AI 学习平台。

## 3. 项目总体架构

项目主要分为两个相对独立但技术上复用的模块：

1. 教材模块拆解与公共知识库构建
2. 用户自主上传 PDF 学习

两者共享底层技术链路：

```text
PDF / Markdown / 笔记
→ 文档解析
→ Markdown 清洗
→ LLM 结构化切分
→ 知识点、例题、解析、练习题、笔记等单元
→ 存储与展示
→ 交互学习 / 知识库管理
```

## 4. 模块一：教材模块拆解与知识库构建

### 4.1 模块目标

将已有数学资料，例如教材 PDF、教师讲义、题库、课堂笔记等，拆解成结构化学习单元，形成可维护、可复用的课程知识库。

这个模块更偏向“平台侧”和“内容生产侧”，关注资料如何被处理、审核、合并和长期维护。

### 4.2 核心流程

1. 上传教材、讲义、题库或已有 Markdown。
2. 使用 MinerU 等工具将 PDF 转换为 Markdown。
3. 使用 LLM 对 Markdown 进行格式修复、断行修复、内容清洗。
4. 使用 LLM 将清洗后的内容拆分为不同类型的学习单元。
5. 将拆分结果存入结构化数据库。
6. 支持用户或队员上传补充笔记。
7. 系统自动判断笔记对应的章节、知识点和内容类型。
8. 笔记进入审核区，审核通过后合并到公共知识库。

### 4.3 内容单元类型

计划支持以下内容类型：

- 章节
- 知识点
- 定义
- 定理
- 公式
- 讲解
- 例题
- 例题解析
- 练习题
- 答案
- 易错点
- 学生笔记
- 教师补充
- 解题技巧

### 4.4 示例数据结构

```json
{
  "course": "高等数学",
  "chapter": "导数与微分",
  "unit": "导数的定义",
  "type": "example",
  "content": "求函数 y = x^2 在 x = 1 处的导数。",
  "solution": "根据导数定义...",
  "tags": ["导数", "极限", "基础题"],
  "difficulty": 1,
  "source": "教材 PDF",
  "status": "approved"
}
```

### 4.5 笔记上传与审核

笔记上传不直接污染主知识库，而是经过以下流程：

```text
上传笔记
→ LLM 判断所属课程、章节、知识点
→ 判断笔记类型：补充讲解 / 例题 / 易错点 / 技巧 / 练习题
→ 进入待审核区
→ 人工审核
→ 合并到对应知识点
```

审核机制的价值：

- 避免错误内容进入公共知识库。
- 保留内容来源和版本记录。
- 支持多人持续贡献。
- 让平台从一次性 demo 变成可扩展系统。

## 5. 模块二：自主上传 PDF 学习

### 5.1 模块目标

允许用户上传自己的 PDF、讲义、笔记或题库，系统自动生成一个个人学习空间。

这个模块更偏向“学生使用侧”和“交互体验侧”，关注用户如何快速开始学习。

### 5.2 核心流程

1. 用户上传 PDF 或 Markdown 文件。
2. 系统自动解析并清洗内容。
3. LLM 提取章节目录和知识点。
4. 系统生成学习路径。
5. 用户进入知识点详情页学习。
6. 用户可以查看 AI 讲解、例题、解析和练习题。
7. 用户可以针对当前资料向 AI 提问。
8. 用户标记掌握情况，系统记录学习进度。

### 5.3 个人学习功能

计划支持以下功能：

- PDF 上传
- 自动生成目录
- 自动生成知识点列表
- 自动总结重点
- 自动提取例题
- 自动生成练习题
- 针对当前资料问答
- 学习进度标记
- 错题记录
- 收藏知识点
- 推荐下一步学习内容

### 5.4 与模块一的关系

自主上传 PDF 生成的内容默认属于个人空间，不直接进入公共教材库。

如果用户上传的笔记或题目质量较高，可以申请贡献到公共知识库，经过审核后进入模块一。

```text
个人 PDF 学习空间
→ 高质量笔记 / 例题 / 解析
→ 申请贡献
→ 审核
→ 公共教材知识库
```

## 6. 当前已有基础

## 8. MVP 范围

为了保证比赛前可以完成，第一版不要做大而全。

建议 MVP 只完成一个完整闭环：

```text
上传一份高等数学 PDF
→ 转 Markdown
→ 清洗 Markdown
→ 拆分出章节、知识点、例题、解析、练习题
→ 在网页中展示
→ 用户针对某个知识点提问
→ 上传一份笔记
→ 自动匹配到对应知识点
→ 审核后合并进知识库
```

MVP 数据范围建议：

- 课程：高等数学
- 章节：极限与连续，或导数与微分
- PDF：1 到 2 份
- 知识点：5 到 10 个
- 每个知识点：1 个讲解、1 道例题、1 个解析、2 道练习题

## 11. 比赛展示故事线

建议演示时按下面流程讲：

1. 学生手里有一份高等数学 PDF 和自己的课堂笔记。
2. 传统情况下，资料分散，复习效率低。
3. 学生上传 PDF。
4. 系统自动解析出章节、知识点、例题和练习题。
5. 系统生成一个可交互的学习空间。
6. 学生点击某个知识点，查看讲解、例题和练习。
7. 学生上传课堂笔记。
8. 系统自动识别笔记属于哪个知识点。
9. 审核通过后，笔记成为该知识点的补充内容。
10. 学生做题后，系统记录掌握情况并推荐下一步学习内容。

核心表达：

> 我们不是把教材搬到线上，而是把学习资料重构成可以互动、补充和复用的学习系统。

## 14. 风险与应对

### 风险一：PDF 解析质量不稳定

应对：

- 先选择排版清晰的 PDF。
- 对公式和图片先做保守处理。
- 比赛展示选择经过验证的样例资料。

### 风险二：LLM 拆分不稳定

应对：

- 设计严格 JSON 输出格式。
- 分章节处理，不一次喂整本书。
- 加入人工审核和修正机制。

### 风险三：范围过大

应对：

- 第一版只做一个章节。
- 优先保证完整流程，而不是覆盖整门课。
- 模块一和模块二都只做最小展示能力。

### 风险四：内容正确性难保证

应对：

- 保留来源。
- 加入审核。
- 让内容测试负责人检查关键样例。
- 对 AI 生成内容标注“AI 生成，需验证”。

## 15. 初步结论

本项目适合交互组，因为它既有明确的 AI 应用场景，也有可展示的用户交互流程。项目的重点不在于从零训练模型，而在于如何把现有资料、PDF 解析、大模型理解和学习交互整合成一个真正可用的学习平台。

第一阶段只要完成“PDF 上传 → Markdown 清洗 → 内容拆分 → 学习界面展示 → 笔记合并”的闭环，就已经具备较强的比赛展示价值。
