# AnyReader UI 可复用能力吸收策略

## 结论

AnyReader UI 的核心是本地优先阅读器和无限画布，而 Owlsome Learning 的核心是公共知识库、私人学习空间和审核贡献闭环。因此不整体迁移 AnyReader 的页面形态，只吸收能增强学习平台的模块化能力。

优先吸收顺序：

1. Markdown 源映射高亮
2. Anchor 锚点系统
3. 问答上下文策略
4. 自动保存状态机
5. 阅读主题系统
6. 无限画布能力作为远期增强

## 可直接吸收的设计

### Markdown 源映射高亮

目标是让用户可以在渲染后的教材里选中一段文字、公式或例题，并把笔记、问答、贡献绑定到原文位置。

Owlsome 中的适用场景：

- 教材段落高亮
- AI 回答引用原文依据
- 笔记绑定到具体定义、定理或例题
- 题目与知识点挂钩时展示证据片段
- 审核者查看贡献来源时快速定位原文

第一版不需要完全实现字符级 source map，可以先在内容单元级别建立锚点，再逐步细化到段落级和字符级。

### Anchor 锚点系统

建议在后续数据模型中统一描述所有“内容绑定关系”：

```text
Anchor
→ public knowledge point
→ content unit
→ personal space
→ personal content unit
→ selected text range
→ optional formula / image / exercise id
```

这将成为笔记、贡献、题目、错题和问答引用的共同基础。

短期落地可以先从前端类型和接口返回字段开始预留，不立刻迁移数据库。

### 问答上下文策略

AnyReader 的上下文模式很完整，Owlsome 可简化为 5 类：

| 模式 | 范围 | 用途 |
|---|---|---|
| 当前知识点 | 一个知识点及其内容单元 | 普通学习问答 |
| 当前章节 | 同章节知识点摘要 | 章节复习 |
| 个人资料 | 个人空间当前知识点 | 私人学习问答 |
| 手动选区 | 用户选中的段落或公式 | 精确解释 |
| 相关资源 | 检索到的笔记、题目、贡献 | 综合回答 |

后续接入 BGE 检索后，“相关资源”模式可由 embedding + reranker 生成。

### 自动保存状态机

适合用于：

- 个人笔记草稿
- 贡献草稿
- 问答历史
- UI 阅读偏好

建议状态：

```text
idle → dirty → saving → idle
                  ↓
                error
```

这部分应先在前端实现为 hook，再决定是否写入后端。

### 阅读主题系统

Owlsome 可以吸收 CSS 自定义属性方案，逐步支持：

- 默认模式
- 阅读模式
- 夜间模式
- 投影展示模式

当前不急于引入完整主题框架，先保持 CSS 变量命名稳定即可。

## 暂不吸收的设计

### 无限画布

无限画布适合探索式阅读和思维整理，但不是 Owlsome 当前的主链路。现阶段优先级低于资源分层、审核权限和内容锚定。

建议远期作为“学习地图”或“知识图谱”视图出现，而不是替换现有主界面。

### Vite 本地文件 API

Owlsome 已有 FastAPI 后端、SQLite 和后续权限需求，不需要再引入 Vite 插件作为本地文件服务器。

### 单页巨组件模式

Owlsome 当前前端已经开始变复杂，应尽快拆分组件，而不是继续集中在一个 `main.tsx` 中。

## 建议后续任务

### Task A：前端组件拆分

目标：把当前单文件前端拆成稳定组件边界。

建议拆分：

- MarkdownRenderer
- AppShell
- Dashboard
- KnowledgeBase
- PersonalSpaces
- ReviewCenter
- SystemOverview

### Task B：内容锚点设计

目标：定义统一 Anchor 类型和第一版接口字段。

第一版只要求锚定到内容单元，不做字符级精确定位。

### Task C：选中文本交互

目标：在公共知识点详情和个人知识点详情中支持选中文本后：

- 提问
- 写笔记
- 申请贡献

### Task D：阅读模式主题

目标：提供更适合长教材阅读的排版和主题切换。

不改变当前功能，只改变阅读体验。
