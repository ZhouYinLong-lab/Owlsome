# Owlsome Learning 项目企划书

面向 Owlsome Team 内部协作与 EL 大赛交互组开发推进。

## 1. 项目背景

数学学习资料通常以 PDF 教材、课程讲义、题库、课堂笔记、错题整理等静态形式存在。它们有三个核心问题：

- 资料难以结构化：PDF 转换后的内容常包含断行、标题错乱、公式混乱、图片路径不稳定等问题。
- 学习过程难以交互：学习者无法围绕某个知识点直接获得讲解、例题、练习、问答和进度反馈。
- 优质笔记难以复用：个人笔记通常停留在个人空间，难以经过审核后沉淀为公共知识库的一部分。

Owlsome Learning 的目标是把静态数学资料转化为可结构化、可审核、可问答、可持续演化的交互式学习空间。

项目当前已有基础：

- `mineru_tools`：PDF/文档解析链路，用于生成 Markdown。
- `text_archiver`：基于 LLM 的 Markdown 清洗与排版工具。
- `owlsome_core`：Obsidian-compatible Markdown 规范化工具。
- `learning_platform`：FastAPI + SQLite + React demo，已支持公共知识库、个人学习空间、笔记审核、问答和进度。

## 2. 项目目标

### 2.1 总体目标

构建一个 AI 交互式数学学习平台，支持从 PDF/Markdown 资料到结构化知识库的自动处理。平台首先保障用户的私人学习空间和资料所有权，在用户主动选择贡献时，再通过审核机制将部分内容沉淀为公共知识库。

### 2.2 核心差异点

Owlsome Learning 不只是“PDF 问答工具”，而是一个同时支持私人学习空间和可选社区共建的数学知识系统。

平台采用“双层知识空间”：

- 私人学习空间：用户上传的资料、笔记、错题和学习进度默认只属于个人，用于个人问答、复习和进度管理。
- 公共知识库：用户主动选择贡献的内容，经自动化处理和审核后，才会进入公共知识库，供后续学习者复用。

核心闭环如下：

```text
个人资料上传
→ 自动解析与清洗
→ 知识点拆分与匹配
→ 生成私人学习空间
→ 私人问答、练习、进度追踪
→ 用户选择是否贡献笔记/题解/讲解
→ 系统自动结构化与预审
→ 审核通过后合并进公共知识库
→ 后续学习者在公共知识库中复用
```

比赛展示故事线：

```text
导入公共教材
→ 上传个人笔记或 Markdown 资料
→ 生成私人学习空间
→ 在私人空间内问答、练习和记录进度
→ 选择优质内容申请贡献到公共知识库
→ 系统自动匹配知识点并预审
→ 审核者通过
→ 公共知识库立即更新
→ 后续问答使用新增社区内容
```

### 2.3 阶段性目标

- Stage 1：完成可稳定演示的本地 MVP。
- Stage 2：完善资料处理链路和 Obsidian-compatible 知识格式。
- Stage 3：建设私人内容到公共知识库的可选贡献、审核和演化闭环。当前已进入实现，第一版闭环已可演示。
- Stage 4：增强 AI 学习交互能力，重点解决知识点、题目和学习行为的关联。
- Stage 5：形成可开源协作、可部署、可维护的完整项目仓库。

### 2.4 用户角色与权限边界

登录系统计划接入南哪小帮手回调，由专人负责实现；本企划书只定义业务权限，不展开登录协议细节。

| 角色 | 权限范围 | 说明 |
|---|---|---|
| 游客 | 浏览公开知识库和演示样例 | 不可上传私人资料，不可提交贡献 |
| 学习者 | 上传私人资料、创建个人学习空间、问答、记录进度、申请贡献 | 默认角色，上传内容默认 private |
| 贡献者 | 在学习者权限基础上，提交笔记、题解、讲解、错题等贡献 | 贡献进入审核队列，不直接进入公共库 |
| 审核者 | 查看待审核贡献、修改匹配知识点、通过、驳回、要求修改 | 可由项目维护者、助教或指定同学担任 |
| 管理员 | 管理课程、公共知识库、审核者权限和系统配置 | 负责公共知识库质量和系统运行 |

权限原则：

- 用户上传资料默认只进入私人空间。
- 只有用户显式申请贡献时，内容才进入贡献审核队列。
- 审核通过前，贡献内容不会污染公共知识库。
- 公共知识库内容需要保留来源、贡献者、审核者和版本记录。

### 2.5 时间线与里程碑

以下时间线以 8 周开发周期为参考，可根据实际比赛截止日期压缩或扩展。

| 周期 | Stage | 关键交付物 | 里程碑验收 |
|---|---|---|---|
| Week 1 | Stage 1 | 本地 MVP 启动、样例知识库导入、基础 README | 前后端可启动，样例导入成功 |

| Week 3 | Stage 2 | Obsidian-compatible 输出与渲染稳定 | callout、wikilink、LaTeX 正常展示 |
| Week 4 | Stage 2 | text_archiver 抽样规范与分段并发原型 | 完成 2 万字以上样例并行清洗测试 |
| Week 5 | Stage 3 | 私人空间、贡献空间、公共库边界和数据模型 | 上传资料默认 private，贡献可进入 pending |
| Week 6 | Stage 3 | 贡献审核页、预审、合并到公共库 | 完成私人内容到公共库的审核闭环 |

Sprint 规划建议：

- 每周一个 Sprint。
- 每个 Sprint 至少包含一个可演示功能、一个技术验证项和一份文档更新。
- 每周末进行一次验收，未达标项进入下一周风险清单。

### 2.6 数据规模与扩展目标

第一版以本地比赛 demo 为目标，不追求大规模并发；但数据模型需要为后续扩展保留空间。

| 阶段 | 用户规模 | 文档规模 | 知识点规模 | 说明 |
|---|---:|---:|---:|---|
| MVP demo | 1-5 名本地演示用户 | 1-3 本教材或讲义 | 50-200 个知识点 | SQLite 足够 |
| 小组内测 | 20-50 名用户 | 20-100 份资料 | 1,000-5,000 个知识点 | 需要基础权限和备份 |
| 课程试点 | 100-500 名用户 | 100-1,000 份资料 | 5,000-50,000 个知识点 | 建议迁移 PostgreSQL |
| 开源社区阶段 | 500+ 用户 | 1,000+ 份资料 | 50,000+ 知识单元 | 需要队列、对象存储和全文/向量检索 |

## 3. 主要阶段与执行路线

## Stage 1：本地 MVP 与比赛展示闭环

目标：形成一个无网络、无 LLM Key 时也可完整演示的第一版平台。

### Phase 1.1 项目仓库整理与基础运行

执行步骤：

1. 保留 `mineru_tools` 与 `text_archiver` 原有能力，不重写工具。
2. 将新增平台代码放入 `learning_platform`。
3. 建立根目录 README、技术文档和版本控制策略。
4. 确保前后端可本地启动。

验收标准：

- 仓库根目录存在清晰 README，包含项目定位、目录说明和启动方法。
- `learning_platform/backend` 可通过 `uvicorn` 启动。
- `learning_platform/frontend` 可通过 `npm run dev` 启动。
- 访问 `/api/health` 返回 `{"ok": true}`。
- 前端页面可打开，无空白页和控制台阻断性错误。

评估方法：

- 本地执行后端健康检查。
- 本地执行前端构建命令。
- 通过浏览器访问页面并完成一次刷新验证。

### Phase 1.2 公共知识库样例导入

执行步骤：

1. 读取已有 MinerU Markdown 样例。
2. 固定导入《微积分 II》第 5 章 `5.1-5.2`。
3. 规则切分章节、知识点、定义、定理、例题、习题。
4. 写入 SQLite 数据库。
5. 重复导入时避免重复插入。

验收标准：

- 点击“一键导入样例”后，数据库至少生成：
  - 1 门课程
  - 1 个章节
  - 6 个以上知识点
  - 30 个以上内容单元
- 重复点击导入按钮不会造成重复数据。
- 前端知识库页面可浏览知识点目录。
- 知识点详情页能展示讲解、定义、例题或习题。

评估方法：

- 调用 `POST /api/import/sample`。
- 调用 `GET /api/stats` 检查数量。
- 手动浏览前端知识库页面。

### Phase 1.3 笔记提交、审核与合并闭环

执行步骤：

1. 在知识点详情页提供笔记提交入口。
2. 系统根据当前知识点或关键词进行匹配。
3. 笔记进入 `pending` 审核队列。
4. 审核通过后，笔记合并到对应知识点。
5. 知识点详情页展示已合并笔记。

验收标准：

- 用户可提交标题和正文不为空的笔记。
- 新笔记默认状态为 `pending`。
- 审核页可看到待审核笔记。
- 点击通过后，笔记状态变为 `approved`。
- 被通过的笔记在知识点详情页可见。
- 点击驳回后，笔记状态变为 `rejected`，不会显示在知识点详情页。

评估方法：

- 调用 `POST /api/notes`。
- 调用 `GET /api/notes/pending`。
- 调用 `POST /api/notes/{id}/approve` 和 `POST /api/notes/{id}/reject`。
- 前端完成一次提交、审核、查看流程。

### Phase 1.4 当前知识点问答

执行步骤：

1. 基于当前知识点内容构造问答上下文。
2. 无 API Key 时使用离线模板回答。
3. 有 OpenRouter API Key 时调用 LLM。
4. 保存问答日志，便于展示交互历史。

验收标准：

- 无 API Key 时，`POST /api/qa` 仍返回可读答案。
- 回答中明确说明“基于当前知识点内容生成”。
- 问答不会因为公式截断导致前端渲染红色错误块。
- `qa_logs` 表中能记录问题、答案和模式。

评估方法：

- 在无 API Key 环境调用问答接口。
- 前端知识点详情页提问一次。
- 检查数据库或 `/api/stats` 中问答记录变化。

## Stage 2：资料处理链路与 Obsidian-compatible 知识格式

目标：让 PDF/Markdown 资料处理结果更加稳定、可复用，并优先兼容 Obsidian 生态。

### Phase 2.1 PDF 转 Markdown 链路标准化

执行步骤：

1. 复用 `mineru_tools` 的 PDF 解析结果。
2. 统一输出 Markdown 文件路径和命名规则。
3. 对生成结果增加 Obsidian-compatible 后处理。
4. 保留原始 Markdown 和规范化 Markdown 两类结果。

验收标准：

- PDF 解析链路不破坏 `mineru_tools` 原有功能。
- 输出 Markdown 至少包含标题、正文、公式和图片引用。
- 规范化结果支持 YAML frontmatter。
- 规范化结果不移除 LaTeX 公式。

评估方法：

- 使用已有样例 PDF 或已有 MinerU 输出进行转换。
- 对比原始 Markdown 与规范化 Markdown。
- 用前端 Markdown 渲染器打开验证。

### Phase 2.2 Markdown 清洗与排版增强

执行步骤：

1. 在 `text_archiver` 中保留 LLM 清洗能力。
2. 提示词要求保护公式、标题、列表、图片和 Obsidian 语法。
3. 无 LLM Key 时允许跳过在线清洗。
4. 清洗后统一调用 `owlsome_core` 规范化。

验收标准：

- 清洗后 Markdown 标题层级更清晰。
- 清洗过程不破坏 `$...$`、`$$...$$`、`\[...\]` 公式。
- `[[双链]]`、`> [!note]`、`==高亮==` 不被错误删除。
- CLI 支持开启或关闭 Obsidian 兼容输出。

评估方法：

- 对同一份 Markdown 执行清洗前后对比。
- 人工抽查至少 5 个公式和 5 个标题。
- 使用前端渲染器验证。

### Phase 2.3 text_archiver 并行清洗与规范抽样

执行步骤：

1. 对整本书先抽样若干代表性片段，覆盖目录、定义、定理、例题、习题、公式密集页和图片页。
2. 使用 LLM 或人工校验抽样片段，生成“本书整理规范”，包括标题层级、公式保留方式、例题格式、习题格式、图片引用格式和 Obsidian callout 使用规则。
3. 将“总规范”和“本书规范”合并为清洗提示词模板。
4. 将长 Markdown 按章节、小节或 token 上限切分为多个可独立处理的片段。
5. 并发调用 text_archiver 清洗多个片段。
6. 对并发结果进行顺序合并、交叉边界检查和全局标题层级修正。
7. 输出清洗报告，记录每个片段耗时、失败次数、重试情况和人工复核建议。

验收标准：

- 在 API 无明显速率限制、片段数不少于 8 个的测试条件下，对同一份样例 Markdown，并行清洗总耗时比串行清洗降低 40% 以上。
- 若 API 存在速率限制，则至少实现失败退避、并发数配置和片段级重试，并输出实际加速比。
- 片段合并后标题顺序与原文顺序一致。
- 片段边界处不出现重复标题、断裂公式或重复段落。
- 每个片段都有处理状态：`pending / running / done / failed / retried`。
- 失败片段可单独重试，不需要重跑整本书。
- 抽样规范中至少包含 6 类规则：标题、公式、例题、习题、图片、Obsidian 语法。

评估方法：

- 选取一份不少于 2 万字、可切分为 8 个以上片段的 Markdown 样例，分别执行串行和并行清洗。
- 统计总耗时、平均片段耗时、失败率和重试次数。
- 记录 API 并发数、速率限制响应和退避策略。
- 人工抽查不少于 10 个片段边界。
- 抽查不少于 10 个公式，确认未被破坏。
- 使用前端渲染器打开合并结果。

### Phase 2.4 Obsidian 渲染与格式兼容

执行步骤：

1. 前端支持 GFM、数学公式、callout、wikilink、highlight。
2. 标题和内容单元标题也支持内联数学公式。
3. 后端问答输出避免截断公式。
4. 文档中明确兼容范围和非目标范围。

验收标准：

- `> [!tip]` 渲染为 callout，不显示原始标记。
- `$D \subseteq R^n$` 在标题和正文中均能渲染。
- `[[知识点]]` 显示为可识别链接文本。
- `==重点==` 显示为高亮。
- 错误公式不会造成页面崩溃。

评估方法：

- 构造包含 Obsidian 语法的测试 Markdown。
- 在知识点详情页和个人空间页分别验证。
- 执行 `npm run build`。

## Stage 3：私人内容的可选贡献与审核闭环

目标：实现项目最大差异点，即用户资料默认保留在私人学习空间，只有在用户主动选择贡献时，相关内容才会经过自动化处理和审核进入公共知识库。

当前状态：第一版已支持从个人知识点申请贡献、进入审核队列、审核通过后合并到公共知识库；驳回和要求修改会保留审核记录且不污染公共库。

### Phase 3.1 私人空间、贡献空间与公共库边界设计

执行步骤：

1. 设计贡献相关数据表。
2. 明确区分私人空间内容、待审核贡献内容和公共知识库内容。
3. 记录贡献者、来源、状态、目标知识点和审核结果。
4. 保留内容版本和审核记录。
5. 为每条私人内容增加可见性字段，默认值为 `private`。

建议数据表：

- `contributors`
- `personal_assets`
- `contributions`
- `contribution_units`
- `contribution_reviews`
- `knowledge_versions`

验收标准：

- 每条贡献都有唯一 ID。
- 每条贡献能追溯到来源资料或用户笔记。
- 每条贡献有明确状态：`draft / pending / approved / rejected / needs_revision`。
- 每条贡献可绑定一个目标知识点或等待匹配。
- 私人内容默认不会出现在公共知识库 API 中。
- 只有用户显式点击贡献后，系统才生成 `contributions` 记录。

评估方法：

- 编写数据库初始化脚本。
- 插入样例贡献数据。
- 查询贡献状态、来源和目标知识点。
- 分别调用私人空间 API 和公共知识库 API，确认数据边界隔离。

### Phase 3.2 个人空间一键申请贡献与上传选项

执行步骤：

1. 在个人知识点详情页加入“申请贡献到公共库”。
2. 用户选择贡献范围：整段知识点、选中片段、笔记、题解、错题或讲解。
3. 用户选择贡献类型：笔记、讲解、例题、习题、易错点。
4. 系统自动提取当前个人知识点内容。
5. 生成待审核贡献。
6. 同时保留“仅上传到私人空间，不贡献公共库”的默认路径。

验收标准：

- 用户可从个人空间创建贡献。
- 贡献进入 `pending` 状态。
- 贡献保留原个人空间 ID 和个人知识点 ID。
- 前端贡献创建后给出明确反馈。
- 上传资料后默认只生成私人学习空间。
- 用户不点击贡献时，公共知识库数据不发生变化。
- 用户点击贡献时，需要看到明确提示：贡献通过审核后将进入公共知识库。

评估方法：

- 上传一份 Markdown 生成个人空间。
- 从个人知识点发起贡献。
- 调用贡献列表 API 检查状态。
- 对比贡献前后的公共知识库内容数量。

### Phase 3.3 自动匹配与预审

执行步骤：

1. 使用关键词、标题、标签和内容相似度匹配公共知识点。
2. 对贡献内容进行类型分类。
3. 检测与已有内容的重复度。
4. 可选使用 LLM 生成预审建议。

验收标准：

- 至少 80% 的样例贡献能匹配到合理知识点。
- 每条贡献至少给出一个推荐知识点。
- 预审结果包含：
  - 推荐知识点
  - 推荐内容类型
  - 重复风险
  - 审核建议
- 无 LLM Key 时仍能通过规则给出基础匹配。

评估方法：

- 准备不少于 10 条样例贡献。
- 人工标注目标知识点。
- 计算自动匹配命中率。
- 对比 LLM 预审和规则预审输出。

### Phase 3.4 审核面板与合并策略

执行步骤：

1. 建设贡献审核页。
2. 展示原文、结构化内容、推荐知识点和预审建议。
3. 审核者可通过、驳回或要求修改。
4. 审核通过后，将贡献合并到公共知识库对应槽位。

公共知识库槽位：

- 教材原文
- 社区讲解
- 精选笔记
- 易错点
- 典型例题
- 习题补充
- FAQ

验收标准：

- 审核者可查看所有 `pending` 贡献。
- 审核者可修改目标知识点和内容类型。
- 审核通过后，公共知识点详情页立即显示新增内容。
- 驳回贡献不会污染公共知识库。
- 所有审核动作写入审核记录。

评估方法：

- 完成一条贡献从创建到审核通过的端到端流程。
- 完成一条贡献从创建到驳回的端到端流程。
- 检查公共知识库是否只出现通过内容。

### Phase 3.5 贡献追踪与社区反馈

执行步骤：

1. 增加贡献者署名。
2. 展示贡献历史。
3. 为公共知识点生成更新记录。
4. 预留点赞、收藏、评论和贡献评分能力。

验收标准：

- 每条公共库社区内容能看到来源贡献。
- 每个贡献者能看到自己的贡献列表。
- 每次合并生成一条知识点更新记录。
- 公共知识点可区分官方教材内容和社区贡献内容。

评估方法：

- 创建多个贡献者样例。
- 审核通过多条贡献。
- 检查贡献者页和知识点更新记录。

## Stage 4：AI 学习交互增强

目标：让平台从资料展示升级为真正的学习辅助系统。

### Phase 4.1 题目与知识点挂钩

执行步骤：

1. 建立题目数据模型，区分教材例题、课后习题、用户错题和社区贡献题目。
2. 为每道题维护题干、答案、解析、来源、难度、标签和关联知识点。
3. 先用规则进行初步挂钩，例如标题编号、章节编号、关键词、公式符号、知识点标签。
4. 对无法明确匹配的题目，使用 LLM 或人工审核生成候选知识点。
5. 支持一道题关联多个知识点，并记录主知识点与辅助知识点。
6. 在知识点详情页展示关联题目，在题目详情页反向展示相关知识点。
7. 用户做错题目后，将错题自动反馈到相关知识点的掌握状态。

验收标准：

- 系统至少支持 4 类题目来源：教材例题、课后习题、用户错题、社区贡献题。
- 每道题至少能关联 1 个知识点。
- 支持一道题关联多个知识点。
- 对人工标注样例集，自动挂钩 Top-1 命中率达到 70% 以上，Top-3 命中率达到 85% 以上。
- 第一轮评估样例不少于 30 道题，仅用于验证算法方向；正式评估样例需扩展到不少于 100 道题，并覆盖至少 5 个小节。
- 知识点详情页能展示不少于 3 道关联题目，若资料不足则明确显示暂无题目。
- 用户将题目标记为“做错”后，相关知识点状态或疑难计数能同步变化。

评估方法：

- 准备不少于 30 道题目的人工标注样例集作为原型评估，并在 Stage 4 完成前扩展到不少于 100 道题。
- 对比自动挂钩结果和人工标注结果。
- 分别报告 Top-1、Top-3 命中率，并说明样本规模局限。
- 在前端完成知识点到题目、题目到知识点的双向跳转。
- 标记 5 道错题，检查相关知识点进度变化。

### Phase 4.2 知识点问答增强

执行步骤：

1. 将问答上下文从单个知识点扩展到相关知识点和关联题目。
2. 引入已审核社区笔记作为回答依据。
3. 引入关联例题、习题和错题作为讲解依据。
4. 回答中标注来源类型。
5. 支持追问。

验收标准：

- 回答能引用当前知识点内容。
- 回答能利用已审核笔记。
- 回答能根据关联题目给出例题式解释。
- 至少支持 3 轮连续追问。
- 无资料依据时明确说明资料不足。

评估方法：

- 对同一知识点进行多轮提问。
- 人工检查回答是否引用无关内容。
- 检查问答日志是否保留上下文。
- 选择 5 道关联题目，确认回答可利用题目解析。

### Phase 4.3 学习进度与个性化建议

执行步骤：

1. 为个人知识点维护掌握状态。
2. 统计学习进度。
3. 根据疑难点和未掌握内容生成学习建议。
4. 关联例题和习题。

验收标准：

- 每个个人知识点有状态：`not_started / learning / mastered / difficult`。
- 个人空间显示进度百分比。
- 状态更新后进度立即变化。
- 系统能根据疑难点生成至少 3 条学习建议。

评估方法：

- 修改多个知识点状态。
- 验证进度统计。
- 调用建议生成接口或前端展示。

### Phase 4.4 练习与错题闭环

执行步骤：

1. 从知识点内容中抽取习题。
2. 用户提交答案或解题笔记。
3. 系统辅助判断解题思路。
4. 错题沉淀为个人笔记或社区贡献候选。

验收标准：

- 每个知识点可展示关联练习题。
- 用户可标记题目状态：未做、做对、做错、需复习。
- 错题可进入个人复习列表。
- 错题讲解可申请贡献到公共库。

评估方法：

- 对至少 5 道习题完成状态流转。
- 检查错题列表。
- 从错题发起一次贡献申请。

## 6. 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| PDF 解析质量不稳定 | 影响知识点拆分 | 保留 Markdown 上传和样例导入作为稳定演示路径 |
| LLM API 不稳定 | 清洗和问答效果波动 | 所有核心流程提供离线规则兜底 |
| text_archiver 串行处理慢 | 整本书清洗耗时过长 | 抽样生成本书规范，分段并发处理，失败片段单独重试 |
| 数学公式渲染复杂 | 前端展示异常 | 使用 KaTeX 容错模式，后端避免截断公式 |
| 用户私人资料被误公开 | 影响数据边界和信任 | 默认 private，只有用户显式申请贡献才进入审核队列 |
| 社区贡献质量参差 | 公共库被污染 | 引入 pending 审核队列、预审和人工确认 |
| 题目和知识点误匹配 | 学习推荐不准确 | 多知识点候选、人工标注样例集评估、审核修正机制 |
| 阶段目标过大 | 开发推进困难 | 每个 stage 拆成可验收 phase，先保证闭环再增强 |

## 7. 建议近期开发优先级

近期建议优先推进 Stage 3，同时把 Stage 2.3 的并行清洗方案作为独立技术任务启动。Stage 3 能体现 Owlsome Learning 的产品差异点，Stage 2.3 则决定整本教材处理能否规模化。

推荐顺序：

1. 明确私人空间、贡献空间和公共知识库的数据边界。
2. 新增贡献数据模型。
3. 将个人空间“申请贡献到公共库”从占位按钮改为真实功能。
4. 建设贡献审核页。
5. 审核通过后合并到公共知识库的不同内容槽位。
6. 为每条社区内容保留来源和审核记录。
7. 设计 text_archiver 抽样规范与分段并发处理方案。
8. 准备题目-知识点挂钩的人工标注样例集。

完成后，项目展示故事线可以升级为：

```text
导入公共教材
→ 上传个人笔记
→ 生成个人学习空间
→ 在私人空间内问答、练习和记录进度
→ 申请贡献到公共知识库
→ 系统自动匹配知识点并预审
→ 审核者通过
→ 公共知识库立即更新
→ 后续问答使用新增社区内容
```

这条线能够清楚表达 Owlsome Learning 的核心价值：用户资料默认服务于个人学习；当用户愿意贡献时，系统再通过自动化处理和审核，把高质量内容沉淀为可复用的公共数学知识库。
