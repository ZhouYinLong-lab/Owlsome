# Owlsome Learning UI 协作指南

## 协作目标

UI 设计的目标不是做一张漂亮首页，而是让 Owlsome Learning 的核心闭环变得清晰、可信、可演示：

```text
公共资源库
→ 私人学习空间
→ 选择性贡献
→ 管理员审核
→ 公共知识库复用
```

设计成员应围绕“学习者”和“管理员”两个角色设计页面，不要把所有功能同时堆在一个控制台里。

## 角色与页面边界

### 学习者

学习者默认只能看到：

- 工作台
- 公共资源库
- 个人学习空间
- 资料处理说明

学习者主要任务：

- 浏览公共教材知识
- 上传或选择个人资料
- 学习知识点并标记进度
- 向当前知识点提问
- 主动申请把个人内容贡献到公共库

### 管理员

管理员额外看到：

- 审核中心
- 系统概览
- 导入样例或管理数据入口

管理员主要任务：

- 审核用户贡献
- 审核普通笔记
- 查看系统统计
- 确认社区内容是否进入公共知识库

当前 demo 使用本地角色切换，正式权限后续接入南哪小帮手登录。

## 设计交付物

每轮 UI 设计至少交付以下内容：

1. 页面线框图
2. 关键状态图
3. 组件清单
4. 交互说明
5. 文案草稿

推荐工具：

- Figma
- 墨刀
- MasterGo
- 或者直接用 Markdown + 截图标注

如果时间紧，优先交付线框图和状态图，不要先追求视觉精修。

## 页面设计清单

### 工作台

必须回答：

- 用户下一步应该做什么？
- 是否已有个人空间？
- 是否有继续学习入口？
- 公共资源库当前有什么？
- 贡献状态是否清楚但不制造压力？

不建议出现：

- 大量数据库统计
- 审核按钮
- 后端技术指标

### 公共资源库

必须体现层级：

```text
学科
→ 教材 / 课程
→ 章节
→ 知识点
→ 内容单元
```

当前 demo 可以固定显示：

```text
数学
→ 微积分 II（第四版）
→ 第 5 章 多元函数微分学
→ 5.1 / 5.2 知识点
```

设计重点：

- 左侧资源树
- 右侧知识点详情
- 教材内容与社区贡献要能区分
- 公式和长文本不能挤压或溢出

### 个人学习空间

必须体现层级：

```text
我的资料
→ 个人空间
→ 个人知识点
→ 进度 / 问答 / 贡献
```

设计重点：

- 上传入口明确
- PDF 当前是接入中，不能暗示已经完成实时解析
- 个人内容默认私有
- 贡献到公共库必须是主动操作

### 审核中心

只给管理员设计。

必须包含：

- 待审核贡献
- 待审核笔记
- 推荐目标知识点
- 匹配理由
- 内容预览
- 通过 / 驳回 / 要求修改

设计重点：

- 审核者能快速判断来源、目标和风险
- 审核通过后进入公共库的结果要可追踪

### 系统概览

只给管理员设计。

可包含：

- 样例导入入口
- 课程数
- 知识点数
- 内容单元数
- 待审核数
- 已合并贡献数
- 社区内容数

不要放在普通学习者首页。

## 组件设计清单

建议优先设计这些组件：

| 组件 | 用途 |
|---|---|
| AppShell | 侧边栏、顶部标题、角色切换 |
| ResourceTree | 学科 / 教材 / 章节 / 知识点树 |
| KnowledgePointCard | 知识点列表项 |
| MarkdownContent | 教材和笔记渲染容器 |
| SourceBadge | 教材内容、清洗版、社区贡献标签 |
| ProgressControl | 未开始 / 学习中 / 已掌握 / 疑难 |
| ContributionForm | 申请贡献表单 |
| ReviewCard | 审核卡片 |
| MetricCard | 管理员统计卡片 |

## 设计规范

### 信息密度

Owlsome 是学习工具，不是营销页面。界面应偏向清晰、克制、可扫描。

推荐：

- 左侧导航 + 右侧内容
- 清晰层级
- 稳定列表和详情布局
- 重要操作靠近当前学习对象

避免：

- 大面积空洞 hero
- 装饰性渐变背景
- 过多统计卡片
- 把管理员功能暴露给普通学习者

### 视觉风格

当前 demo 使用偏阅读和知识库的低饱和配色：

- 深绿黑：侧边栏和主要文字
- 金色：主要按钮
- 浅绿灰：辅助背景
- 白色：内容卡片

新设计可以调整，但需保持：

- 阅读友好
- 数学公式清晰
- 长文本不压迫
- 组件边界明确

### 响应式

至少考虑：

- 1440px 桌面演示
- 1280px 笔记本
- 768px 平板或窄屏

比赛 demo 优先保证桌面端。

## 协作流程

### Step 1：确定页面目标

每个页面先写一句话目标。例如：

```text
公共资源库页面用于帮助学习者按教材层级找到知识点，并查看教材内容、社区贡献和问答入口。
```

### Step 2：交付低保真线框

先画布局，不上颜色。

必须标注：

- 页面区域
- 主要操作按钮
- 数据从哪里来
- 空状态如何显示

### Step 3：产品和技术一起评审

评审只看四点：

- 是否符合角色边界
- 是否支撑核心闭环
- 是否能用现有 API 实现
- 是否比当前界面更清晰

### Step 4：再做视觉稿

线框确认后再做颜色、字体、间距和图标。

### Step 5：开发落地

开发不应凭感觉补设计缺口。若设计稿缺少状态，需要回到设计阶段补齐。

## 给设计成员的任务模板

可以直接复制给队友：

```text
请为 Owlsome Learning 设计【页面名称】。

目标用户：
页面目标：
必须包含：
不能包含：
关键状态：
参考数据：
交付格式：
截止时间：
```

示例：

```text
请为 Owlsome Learning 设计“公共资源库”页面。

目标用户：学习者
页面目标：按学科、教材、章节找到知识点，并阅读教材内容与社区贡献。
必须包含：资源树、知识点列表、知识点详情、来源标签、问答入口。
不能包含：审核操作、系统统计、导入按钮。
关键状态：未导入数据、已导入 8 个知识点、存在社区贡献。
参考数据：数学 / 微积分 II（第四版）/ 第 5 章 / 5.1-5.2。
交付格式：Figma 或截图标注。
截止时间：YYYY-MM-DD HH:mm。
```

## 验收标准

一份 UI 设计稿可以进入开发，需要满足：

- 页面目标明确
- 学习者和管理员功能没有混在一起
- 至少覆盖正常状态和空状态
- 关键按钮和交互路径明确
- 文案不误导当前能力
- 能用现有 API 或明确标注为未来能力

如果设计稿只是“好看”，但无法说明数据来源和状态变化，不进入开发。
